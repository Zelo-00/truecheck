ЧАСТЬ I. ФУНДАМЕНТ: ВАШ СЕРВЕР

Любая серьёзная работа с ИИ-агентами рано или поздно упирается в один вопрос: где это всё будет работать? Агент, который крутится только в окне вашего ноутбука, умирает, как только вы закрываете крышку. Чтобы агенты работали круглосуточно, принимали вебхуки, обрабатывали очереди задач и были доступны из интернета, им нужен сервер. В этой части мы с нуля подготовим такой сервер — VPS — и сделаем это безопасно.

Модуль 1. Что такое VPS и зачем он AI-инженеру

Чему вы научитесь: понимать, что такое VPS, чем он отличается от облака и собственного компьютера, и как выбрать подходящий тариф под задачи ИИ-агентов.

Зачем это нужно

Представьте, что вы построили агента, который раз в час проверяет почту, классифицирует письма дешёвой LLM и складывает важные в Telegram. На вашем ноутбуке он будет работать ровно до обеда. На сервере — годами. VPS (Virtual Private Server, виртуальный частный сервер) — это, по сути, арендованный компьютер в дата-центре, который включён всегда, имеет постоянный белый IP-адрес и доступен по сети из любой точки мира.

Теория: VPS, выделенный сервер и облако

Чтобы не путаться в терминах, разложим по полочкам три способа «получить вычисления в аренду»:

Для целей курса — обучение, личные агенты, n8n, небольшой каскад — обычный VPS оптимален: предсказуемая цена, полный контроль, и вы реально учитесь администрировать систему, а не прятаться за кнопками облачной панели.

Как выбрать тариф

Ключевые параметры VPS и что они значат для ИИ-агентов:

  • vCPU (ядра процессора). Агенты сами по себе почти не грузят процессор: тяжёлые вычисления происходят на стороне LLM-провайдера. 2 ядра достаточно для n8n + несколько Docker-контейнеров. Процессор станет узким местом, только если вы решите запускать локальные модели на CPU (медленно) — но для этого правильнее брать GPU.
  • RAM (оперативная память). Самый важный параметр для нашего сценария. n8n, база данных, пара агентов-сервисов — это минимум 2 ГБ, комфортно 4 ГБ. Если планируете локальные модели через Ollama даже небольшого размера — 8–16 ГБ.
  • Диск (SSD/NVMe). 20–40 ГБ хватает для системы, Docker-образов и логов. NVMe предпочтительнее: быстрее стартуют контейнеры и базы.
  • Трафик. Агенты гоняют в основном текст (запросы к API) — это копейки трафика. Лимиты в десятки ТБ вы не выберете.
  • Расположение дата-центра. Влияет на задержку (ping). Берите регион ближе к себе и к API, которые используете. Для российских пользователей популярны отечественные провайдеры (например, Timeweb, Selectel, Reg.ru, RuVDS).
Пример из практикиПример из практики

Сервер, на котором готовился и развёрнут этот курс, — обычный Ubuntu-VPS, поднятый с нуля по шагам из этой части (тот самый сервер, с которого, возможно, вы и читаете веб-версию курса). На нём одновременно живут несколько контейнеров Docker за обратным прокси nginx: веб-приложения на разных портах, агенты и боты в отдельных контейнерах, плюс Claude Code, Python-окружения и Node через nvm. Конфигурации 2 vCPU / 4 ГБ RAM / NVMe для такого набора достаточно; апгрейд у большинства провайдеров делается в пару кликов без переустановки. Дальше в этой части — ровно те шаги, которыми этот сервер был приведён из «голой» машины в рабочее состояние.

Выбор операционной системы

При создании VPS провайдер предложит «образ» (image) — готовую ОС. Берите Ubuntu LTS (на момент курса актуальны 22.04 и 24.04 LTS). Причины:

  • LTS = Long Term Support, обновления безопасности на 5 лет.
  • Огромное сообщество: на любую ошибку уже есть готовый ответ.
  • Docker, Node.js, Python ставятся штатно и без сюрпризов.

Debian — тоже отличный, более «пуристский» выбор. CentOS/RHEL-семейство и экзотику новичку брать не стоит: меньше готовых инструкций.

Упражнения

  1. Зайдите на сайт двух любых VPS-провайдеров и сравните минимальные тарифы по RAM и цене. Прикиньте, на каком вы развернёте n8n + 2 агента. Признак успеха: вы выписали два тарифа с RAM ≥ 4 ГБ и можете объяснить, почему RAM для агентов важнее CPU.

Модуль 2. Первое подключение и базовая настройка

Чему вы научитесь: подключаться к свежему серверу по SSH, обновлять систему, создавать непривилегированного пользователя и настраивать вход по ключу вместо пароля.

Зачем это нужно

Когда провайдер выдаёт VPS, вы получаете «голую» машину с пользователем root и паролем. В таком состоянии сервер уязвим: боты по всему интернету круглосуточно перебирают пароли к стандартному SSH-порту. Первые 15 минут жизни сервера — самые важные. Мы превратим уязвимую машину в защищённую: вход только по ключу, работа из-под обычного пользователя, закрытые порты.

Шаг 1. Подключение по SSH

SSH (Secure Shell) — протокол для безопасного удалённого управления сервером через зашифрованный канал. После создания VPS провайдер пришлёт вам три вещи: IP-адрес, имя пользователя (обычно root) и пароль.

На Linux/macOS откройте терминал. На Windows используйте встроенный ssh в PowerShell или Windows Terminal (он есть начиная с Windows 10):

ssh root@<ВАШ_IP>

При первом подключении SSH спросит, доверяете ли вы «отпечатку» сервера (fingerprint) — отвечайте yes. Затем введите пароль (символы при вводе не отображаются — это нормально). Если всё верно, вы увидите приглашение вида root@hostname:~#.

ЗаметкаНестандартный порт

Некоторые провайдеры сразу выдают SSH не на стандартном порту 22, а на другом — для защиты от ботов. Тогда подключение выглядит так: ssh -p <ПОРТ> root@<ВАШ_IP>. Уточните порт в письме провайдера или панели управления.

Пример из практикиПример из практики

На этом самом сервере наружу опубликован только обратный прокси nginx, а сами приложения слушают внутренние порты в docker-сети и напрямую из интернета недоступны. Это пример «эшелонированной обороны»: даже если в приложении ошибка, снаружи к нему не достучаться — виден лишь nginx, который вы контролируете.

Шаг 2. Обновление системы

Первое, что делают на новом сервере, — обновляют все пакеты. В свежем образе наверняка есть устаревшие версии с известными уязвимостями.

apt update && apt upgrade -y
  • apt update — обновляет список доступных пакетов (не сами пакеты).
  • apt upgrade -y — устанавливает обновления; -y автоматически отвечает «да».

Если в процессе ядро обновится, в конце может потребоваться перезагрузка:

reboot

После reboot соединение разорвётся — подождите минуту и подключитесь снова.

Шаг 3. Создание непривилегированного пользователя

Работать постоянно под root — плохая практика: любая ошибочная команда (или вредоносный скрипт) выполняется с максимальными правами и может уничтожить систему. Правильный подход — завести обычного пользователя и давать ему права администратора только по необходимости через sudo.

Создаём пользователя (замените deploy на любое имя):

adduser deploy

Команда спросит пароль и пару необязательных полей (имя и т.п. — можно пропустить, нажимая Enter). Теперь дадим пользователю право выполнять команды от администратора:

usermod -aG sudo deploy

Здесь -aG sudo добавляет (-a) пользователя в группу (-G) sudo. Члены этой группы могут предварять команды словом sudo, чтобы выполнить их с правами root.

СоветПочему именно deploy

Имя пользователя — на ваш вкус, но избегайте предсказуемых admin, user, test: их боты перебирают в первую очередь. Имя вроде deploy или ваше собственное — нормальный выбор.

Шаг 4. Вход по SSH-ключу вместо пароля

Это — ключевое улучшение безопасности. SSH-ключ — это пара файлов: приватный (остаётся у вас на компьютере, его нельзя никому показывать) и публичный (кладётся на сервер). Сервер пускает только того, у кого есть приватный ключ, соответствующий публичному. Подобрать такой ключ перебором практически невозможно, в отличие от пароля.

На вашем локальном компьютере (не на сервере!) сгенерируйте ключ, если у вас его ещё нет:

ssh-keygen -t ed25519 -C "my-vps-key"
  • -t ed25519 — современный, быстрый и стойкий тип ключа (предпочтительнее устаревшего RSA).
  • -C — комментарий-метка, чтобы потом узнать ключ.

На вопрос о расположении файла нажмите Enter (примет путь по умолчанию ~/.ssh/id_ed25519). На вопрос про passphrase — желательно задать пароль на сам ключ (это вторая линия защиты, если ваш ноутбук украдут).

Теперь скопируйте публичный ключ на сервер. Проще всего командой:

ssh-copy-id deploy@<ВАШ_IP>

Если ssh-copy-id недоступен (например, на Windows), скопируйте вручную. Покажите содержимое публичного ключа на локальной машине:

cat ~/.ssh/id_ed25519.pub

Скопируйте строку, затем на сервере (под пользователем deploy) выполните:

mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "<ВСТАВЬТЕ_ПУБЛИЧНЫЙ_КЛЮЧ>" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

Права 700 и 600 критичны: SSH откажется использовать ключи, если к каталогу или файлу есть доступ у посторонних. Проверьте вход с локальной машины:

ssh deploy@<ВАШ_IP>

Если вошли без запроса пароля сервера (возможно, спросит passphrase ключа) — успех.

Шаг 5. Запрет входа по паролю и под root

Теперь, когда вход по ключу работает, отключим вход по паролю — это закроет 99% автоматических атак. Откройте конфиг SSH:

sudo nano /etc/ssh/sshd_config

Найдите и приведите к такому виду строки (уберите # в начале, если есть):

PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
  • PermitRootLogin no — запрет прямого входа под root.
  • PasswordAuthentication no — запрет входа по паролю (только ключи).

Сохраните (в nano: Ctrl+O, Enter, затем Ctrl+X) и перезапустите SSH:

sudo systemctl restart ssh
ВниманиеНе закройте себе дверь

Не закрывайте текущую SSH-сессию, пока не проверите вход в новом окне! Откройте второй терминал и убедитесь, что ssh deploy@<ВАШ_IP> работает по ключу. Если что-то пошло не так, у вас останется первая рабочая сессия, чтобы откатить изменения. Это правило спасало многих от полной потери доступа к серверу.

Модуль 3. Сеть, файрвол и защита от перебора

Чему вы научитесь: настраивать файрвол ufw, осознанно открывать только нужные порты, при желании менять SSH-порт и ставить fail2ban против ботов.

Зачем это нужно

Сервер в интернете — как дом на оживлённой улице. Файрвол — это решётка на окнах: он определяет, через какие «двери» (порты) вообще можно стучаться. По умолчанию свежий сервер часто принимает соединения на множество портов. Наша задача — оставить открытыми только те, что реально нужны, и закрыть всё остальное.

Теория: что такое порты

Когда программа на сервере ждёт соединений, она «слушает» определённый порт — число от 1 до 65535. Несколько важных для нас:

  • 22 — SSH (управление сервером).
  • 80 — HTTP (веб без шифрования).
  • 443 — HTTPS (веб с шифрованием).
  • 5678 — n8n (по умолчанию).
  • 5432 — PostgreSQL (база данных — её наружу открывать НЕ нужно).

Принцип безопасности — «запрещено всё, что не разрешено явно» (default deny). Мы закроем входящие соединения по умолчанию и точечно откроем только SSH и веб.

Шаг 1. Настройка ufw

ufw (Uncomplicated Firewall) — простая надстройка над системным файрволом Linux. Установим (в Ubuntu обычно уже есть) и настроим:

sudo apt install ufw -y

# Политики по умолчанию: входящие — запретить, исходящие — разрешить
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Разрешаем то, что нужно
sudo ufw allow OpenSSH        # или: sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
ВниманиеСначала разрешите SSH!

Если вы включите файрвол, не разрешив SSH, вы немедленно потеряете доступ к серверу. Всегда сначала allow OpenSSH, и только потом enable.

Включаем файрвол и проверяем:

sudo ufw enable
sudo ufw status verbose

Вы увидите список разрешённых портов. Всё остальное теперь закрыто для входящих соединений.

Шаг 2 (опционально). Смена SSH-порта

Перенос SSH с порта 22 на нестандартный (например, 2222) — не «настоящая» защита (стойкость даёт запрет паролей), но резко снижает шум в логах от ботов. В /etc/ssh/sshd_config поменяйте Port 22 на свой, затем откройте новый порт в ufw до перезапуска SSH:

sudo ufw allow 2222/tcp
sudo systemctl restart ssh

Подключаться теперь нужно с -p 2222. Старое правило для 22 можно удалить: sudo ufw delete allow 22/tcp.

Пример из практикиПример из практики

На этом сервере доступ по SSH защищён ключами и сужен, а наружу торчит только обратный прокси nginx — сами приложения сидят на внутренних портах docker-сети. ufw на хосте дублирует эту защиту. Дублирование (defense in depth) — норма в безопасности: одна ошибка не должна открывать весь сервер.

Шаг 3. fail2ban против перебора

fail2ban автоматически банит IP-адреса, которые слишком часто ошибаются паролем или подозрительно стучатся. Даже при входе по ключу он полезен — отсекает назойливых ботов.

sudo apt install fail2ban -y
sudo systemctl enable --now fail2ban

По умолчанию он уже защищает SSH. Проверить статус:

sudo fail2ban-client status sshd

Чеклист модуля 3

  • Установил и настроил ufw с политикой default deny.
  • Открыл только SSH, 80 и 443.
  • (Опц.) Перенёс SSH на нестандартный порт.
  • Поставил fail2ban.

Упражнения

  1. Выведите sudo ufw status numbered и удалите одно лишнее правило по номеру.
  2. Посмотрите в логах SSH (sudo journalctl -u ssh | tail -50), сколько было попыток входа за день.

Модуль 4. Docker, swap и эксплуатация сервера

Чему вы научитесь: устанавливать Docker, понимать зачем он нужен агентам, добавлять swap-память и базово следить за здоровьем сервера.

Зачем это нужно

Почти всё, что мы будем разворачивать дальше — n8n, базы данных, агенты-сервисы — удобнее всего запускать в Docker. Docker упаковывает приложение со всеми зависимостями в изолированный «контейнер», который одинаково работает на любом сервере. Это избавляет от классической боли «у меня на машине работало, а на сервере нет».

Теория: контейнеры простыми словами

Контейнер — это лёгкая изолированная среда, где приложение видит только свои файлы и библиотеки. В отличие от виртуальной машины, контейнер не тащит за собой целую ОС, поэтому стартует за секунды и потребляет мало ресурсов. Для нас важны три понятия:

  • Образ (image) — «слепок» приложения (например, n8nio/n8n). Скачивается из реестра (Docker Hub).
  • Контейнер (container) — запущенный экземпляр образа.
  • Том (volume) — постоянное хранилище, которое переживает пересоздание контейнера (туда кладут базы и настройки).

Шаг 1. Установка Docker

Официальный скрипт установки:

curl -fsSL https://get.docker.com | sudo sh

Чтобы запускать Docker без sudo, добавьте своего пользователя в группу docker и перелогиньтесь:

sudo usermod -aG docker $USER
# выйдите и зайдите снова по SSH, чтобы группа применилась

Проверка:

docker run hello-world

Если увидели приветствие — Docker работает. Также сразу пригодится docker compose (входит в современный Docker как плагин):

docker compose version

Для одного контейнера годится docker run. Но как только сервисов становится несколько (n8n + база), удобнее описать их в файле docker-compose.yml и поднимать одной командой docker compose up -d. Мы будем активно этим пользоваться в части про n8n.

Шаг 2. Добавление swap

Swap — это файл на диске, который система использует как «запасную» оперативную память, когда настоящей не хватает. На VPS с 2–4 ГБ RAM swap спасает от падения процессов при пиковых нагрузках (например, сборка Docker-образа или всплеск активности агентов).

Создадим swap-файл на 2 ГБ:

sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

Чтобы swap включался после перезагрузки, добавим его в /etc/fstab:

echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

Проверка:

free -h

В строке Swap должно появиться 2 ГБ.

ЗаметкаSwap — не замена RAM

Диск в десятки раз медленнее оперативной памяти. Swap — это «подушка безопасности», а не способ сэкономить на RAM. Если сервер постоянно сидит в swap — пора увеличивать тариф.

Шаг 3. Базовый мониторинг

Несколько команд, которые AI-инженер должен знать наизусть:

htop            # интерактивный монитор процессов (sudo apt install htop)
df -h           # сколько места на диске
free -h         # память и swap
docker ps       # запущенные контейнеры
docker stats     # потребление ресурсов контейнерами в реальном времени
journalctl -f   # «живой» системный лог

Привычка раз в день смотреть df -h (не забился ли диск логами) и docker ps (всё ли живо) экономит часы расследований.

Модуль 5. Секреты, ключи и гигиена доступа

Чему вы научитесь: безопасно хранить API-ключи и пароли, не утекать секреты в git и публичные файлы, организовать .env-файлы.

Зачем это нужно

ИИ-агенты живут на API-ключах: OpenRouter, OpenAI, Telegram-бот, база данных. Каждый такой ключ — это деньги и доступ. Утёкший ключ от OpenRouter может за ночь списать весь баланс, а утёкший ключ от сервера — отдать машину злоумышленнику. Этот модуль — про дисциплину, которая отличает профессионала.

Правило 1. Секреты — в переменных окружения, не в коде

Никогда не пишите ключи прямо в коде или в командах. Храните их в файле .env:

OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxxxxxxxxx
TELEGRAM_BOT_TOKEN=1234567:AAabc...
DATABASE_URL=postgresql://user:pass@localhost:5432/mydb

Приложение читает их как переменные окружения. В Python:

import os
api_key = os.environ["OPENROUTER_API_KEY"]

Правило 2. .env — никогда не в git

Файл .env должен быть в .gitignore, чтобы случайно не закоммитить ключи в репозиторий (особенно публичный на GitHub — там их за минуты находят боты-сканеры):

# .gitignore
.env
*.key
*.pem
secrets/

Для коллег и будущего себя оставляйте .env.example — тот же файл, но с пустыми значениями:

OPENROUTER_API_KEY=
TELEGRAM_BOT_TOKEN=
ВниманиеЕсли ключ всё-таки утёк

Удаление файла из репозитория не помогает — ключ остаётся в истории git и в кэшах. Единственно верное действие: немедленно отозвать (revoke) ключ в панели провайдера и выпустить новый. Считайте утёкший ключ скомпрометированным навсегда.

Правило 3. Права доступа к файлам секретов

Ограничьте чтение .env только владельцем:

chmod 600 .env

600 означает: владелец может читать и писать, остальные — ничего.

Правило 4. Минимум привилегий

  • Для каждого сервиса — отдельный ключ (чтобы при утечке отозвать только его).
  • На OpenRouter и подобных — выставляйте лимит расхода на ключ.
  • Базу данных не открывайте в интернет: пусть слушает только localhost или внутреннюю docker-сеть.
  • Не давайте агентам прав больше, чем нужно для их задачи (об изоляции прав — отдельно в части про агентов).

Правило 5. Резервные копии

Перед любой рискованной операцией делайте бэкап. Простейший способ — архив каталога проекта:

tar -czf backup-$(date +%Y%m%d-%H%M%S).tar.gz /путь/к/проекту

Базы данных бэкапятся своими средствами (для PostgreSQL — pg_dump). Привыкайте: бэкап стоит секунды, восстановление из небытия — невозможно.

Чеклист модуля 5

  • Все ключи лежат в .env, а не в коде.
  • .env добавлен в .gitignore, есть .env.example.
  • На файлы секретов стоит chmod 600.
  • На API-ключах выставлены лимиты расхода.
  • Понимаю, что утёкший ключ нужно отзывать, а не «прятать».

Упражнения

  1. Создайте в учебном проекте .env, .env.example и .gitignore. Убедитесь, что git status не показывает .env.
  2. Найдите в панели вашего LLM-провайдера, где выставляется лимит расхода на ключ, и поставьте небольшой лимит.
