Модуль 10. Что такое ИИ-агент

Чему вы научитесь: объяснять архитектуру агента на уровне компонентов и понимать цикл «рассуждение — действие — наблюдение».

Зачем это нужно

Слово «агент» затёрто маркетингом. Чтобы строить надёжные системы (и отличать настоящего агента от чат-бота с красивым названием), нужна чёткая ментальная модель. Без неё вы будете «лечить симптомы»; с ней — проектировать.

Определение через четыре компонента

ИИ-агент = языковая модель (LLM) + инструменты + память + цикл. Разберём:

  1. LLM (модель) — «мозг». Принимает текст, возвращает текст. Сама по себе модель ничего не умеет делать в мире — только рассуждать словами.
  2. Инструменты (tools) — «руки». Это функции, которые модель может вызвать: прочитать файл, выполнить команду, сходить в API, записать в БД. Модель не выполняет их сама — она просит харнесс (программную обвязку-среду вокруг модели) выполнить, получает результат и продолжает.
  3. Память (memory) — хранилище. Контекст текущего диалога (краткосрочная) и сохранённые между сессиями знания — файлы, заметки, база (долгосрочная).
  4. Цикл (loop) — «воля». То, что заставляет агента не остановиться на одном ответе, а продолжать: рассуждать → действовать → наблюдать результат → снова рассуждать, пока задача не решена.
ЗаметкаКлючевое отличие агента от чат-бота

Чат-бот: один запрос → один ответ. Агент: одна задача → много шагов, инструментов и самопроверок до результата. Если система сама решает, какие действия предпринять, и доводит дело до конца — это агент.

Цикл ReAct: рассуждение + действие

Базовый паттерн работы агента называется ReAct (Reasoning + Acting). На каждом шаге модель:

  1. Рассуждает (Thought): «Чтобы ответить, мне нужно найти файл конфигурации».
  2. Действует (Action): вызывает инструмент grep config.
  3. Наблюдает (Observation): получает результат — список файлов.
  4. Повторяет, пока не сделает вывод (Final Answer).
        ┌─────────────────────────────────────────────┐
        │                ЗАДАЧА                        │
        └───────────────────┬─────────────────────────┘
                            ▼
              ┌──────── Thought (рассуждение) ◀────────┐
              │   «что мне нужно сделать дальше?»       │
              ▼                                         │
        Action (действие) ── вызов инструмента          │
              │            (Read / Bash / Grep …)       │
              ▼                                         │
        Observation (наблюдение) ── результат ──────────┘
              │
              ▼  (цикл повторяется, пока задача не решена)
        Final Answer (вывод)

Это и есть «агентный цикл». В Claude Code он реализован за вас: модель сама чередует размышления и вызовы инструментов (Read, Bash, Edit и т.д.), а харнесс исполняет действия и возвращает наблюдения.

Тулзы: как модель вызывает инструменты

Технически инструмент описывается схемой: имя, описание (когда применять), параметры. Модель, видя задачу, генерирует «вызов инструмента» — структурированный объект с именем и аргументами. Харнесс исполняет его и возвращает результат как «наблюдение». Цикл продолжается, пока модель не перестанет вызывать инструменты (признак завершения — «end_turn»).

СоветОписание инструмента — это промпт

Модель решает, вызывать ли инструмент, по его описанию. Хорошее описание прескриптивно: не только «что делает», но и «когда вызывать» — «Вызови это, когда пользователь спрашивает про текущие цены». Современные модели берут инструмент аккуратнее, поэтому триггер в описании ощутимо повышает «попадание».

Планирование и рефлексия

Поверх ReAct надстраиваются два важных паттерна:

  • Планирование (planning) — агент сначала разбивает большую задачу на план из шагов, потом исполняет. Уменьшает «блуждание» на сложных задачах. В Claude Code этому служит plan mode.
  • Рефлексия (reflection) — агент критикует собственный результат и улучшает его. Часто эффективнее, когда критику делает отдельный агент со «свежим контекстом», а не сам автор (он «влюблён» в своё решение).

Память: краткосрочная и долгосрочная

  • Краткосрочная — окно контекста. Ограничено (хоть и велико — до 1М токенов у топ-моделей). Когда оно переполняется, применяют компактизацию (сжать историю в резюме) или редактирование контекста (выкинуть устаревшие результаты инструментов).
  • Долгосрочная — файлы/заметки/БД. Агент пишет «себе на будущее»: что сработало, какие решения приняты. На следующей сессии читает это и не повторяет ошибок. Простейшая реализация — .md-файл с одной мыслью на запись.

Когда НЕ строить агента

Зрелость инженера — знать, когда агент избыточен. Проверьте четыре критерия: сложность (задача многошаговая и плохо формализуемая?), ценность (результат оправдывает цену и задержку?), выполнимость (модель в принципе справляется?), цена ошибки (ошибку можно поймать и откатить?). Если на любой ответ «нет» — хватит одного запроса к модели или простого workflow, а не агента.

  • Понимаю цикл ReAct (Thought → Action → Observation).

  • Знаю 4 критерия «строить ли агента».

  2. Приведите пример задачи, для которой агент избыточен, и объясните почему.

Модуль 11. Оркестрация: топологии многоагентных систем

Чему вы научитесь: выбирать архитектуру взаимодействия нескольких агентов и грамотно передавать между ними контекст.

Зачем это нужно

Один агент с одним огромным промптом — хрупок: он перегружается, путается, дорого стоит. Оркестрация — это искусство разбить работу между специализированными агентами так, чтобы система была надёжнее, дешевле и понятнее. Это центральный навык AI-инженера уровня senior.

Почему несколько агентов лучше одного «супер-промпта»

Вспомните методику KSCR (модуль 13): когда в один промпт пихают слишком много требований, модель не удерживает их все в «рабочей памяти» — растёт разрыв между знанием и умением (когнитивная перегрузка). Лечение — разбить на подзадачи. Многоагентность — это разбиение, доведённое до архитектуры: каждый агент решает узкую задачу с высокой компетентностью, а оркестратор собирает результат.

Топологии оркестрации

Три самые ходовые топологии наглядно:

  [Исследователь] ─▶ [Черновик] ─▶ [Ревью] ─▶ [Финал]

                 ┌─▶ [Субагент: поиск]

                 └─▶ [Субагент: писатель]

КАСКАД (экономия):
  [Дешёвый роутер] ─▶ [Дешёвые воркеры] ─▶ [Дешёвый судья]
                                              │ провал?
                                              ▼
                                    [Сильная модель] (редко)

Каскад как способ экономии (ключевой паттерн курса)

Каскад заслуживает отдельного разбора, потому что это и архитектура, и стратегия удешевления. Идея: 90% работы — это рутинные операции (классификация, извлечение, черновики), которые отлично делает дешёвая модель. И только 10% — трудные шаги (план, спорное решение, финальная проверка), которые стоит отдать сильной модели.

Типовые роли каскада и какие модели на них ставить (подробнее — в части про LLM):

  • Роутер/классификатор — самая дешёвая модель: определить тип запроса, выбрать ветку.
  • Воркер (массовая обработка) — дешёвая модель: извлечь, преобразовать, набросать черновик.
  • Суммаризатор — дешёвая, но аккуратная модель: сжать длинный контекст.
  • Планировщик/оркестратор — сильная модель: разбить задачу, выбрать инструменты, рассуждать.
  • Критик/судья — аккуратная модель: проверить результат воркера, оценить качество.

Стратегия «дешёвый черновик → дорогая проверка»: воркер на дешёвой модели генерирует кандидат, дешёвый «судья» проверяет, и только при провале проверки задача эскалируется на сильную модель. Так большинство запросов вообще не доходит до дорогой модели.

Передача контекста (handoff) — где всё ломается

Слабое место любой многоагентной системы — передача контекста между агентами. Агент-исполнитель знает только то, что ему передали. Правила здравой передачи:

  • Передавайте цель и намерение, а не только задачу: «Я готовлю отчёт для клиента X, ему нужно Y; с учётом этого — сделай Z». Модель свяжет задачу с релевантной информацией.
  • Передавайте результат в сжатом, структурированном виде, а не «сырой дамп» всего, что наработал предыдущий агент.
  • Изолируйте контекст: тяжёлую работу (чтение 30 файлов) делает субагент в своём окне, наверх отдаёт короткий итог.
  • Для долгих процессов давайте агентам общую память (файл/БД), чтобы контекст переживал передачи.
ВниманиеГлавная ошибка оркестрации

Не предполагайте, что субагент «помнит» то, что обсуждал оркестратор. У него отдельный контекст. Если субагенту нужно знать предысторию — её нужно явно передать в задаче или записать в общую память. «Молчаливое предположение об общем контексте» — причина большинства сбоев каскадов.

Асинхронная оркестрация

Сильные современные модели надёжно работают с асинхронными субагентами: оркестратор делегирует независимые подзадачи и продолжает работать, не блокируясь на самом медленном. Долгоживущие субагенты сохраняют свой контекст между подзадачами (экономия на повторном «разогреве»), а оркестратор не простаивает. Это будущее каскадов: не «запустил и жду», а «делегировал и работаю дальше».

Модуль 12. Изолированные правила: как проектировать «правила игры» для агентов

Чему вы научитесь: придумывать изолированные правила, понимать принцип изоляции и разбирать реальный пример — трёхкомпонентную архитектуру (ТКОП — трёхкомпонентный оптимизатор промптов).

Зачем это нужно

Агент без правил — как сотрудник без должностной инструкции: делает что хочет, путает роли, выходит за границы полномочий. Изолированные правила задают агенту чёткую роль, границы и протокол. «Изолированные» — потому что каждый агент видит только свой свод правил и работает только со своими данными, не имея доступа к чужому контексту. Это делает систему предсказуемой, безопасной и отлаживаемой.

Что такое изоляция и зачем она

Изоляция — это принцип, по которому агент:

  • знает только свою роль и свои полномочия;
  • работает только с переданными ему данными (копиями), не имея доступа к среде других;
  • не может выйти за границы своих прав, даже если его об этом «попросят».

Зачем: безопасность (узкий «радиус поражения» при ошибке или промпт-инъекции), предсказуемость (поведение детерминировано ролью), отлаживаемость (понятно, кто за что отвечает). Технически изоляция достигается через: отдельные системные промпты, ограничение инструментов (минимум прав), отдельные окна контекста, sandbox/Docker для исполнения.

Как придумывать изолированные правила: рецепт

Хорошее правило для агента отвечает на вопросы (заметьте, это перекликается с чеклистом полноты промпта из KSCR):

  1. Роль и миссия. Кто этот агент и какова его единственная цель? («Ты — АССИСТЕНТ, твоя роль исключительно аналитическая».)
  2. Полномочия. Что агенту разрешено делать? (анализировать, планировать, генерировать варианты).
  3. Запреты. Что категорически нельзя? (прямой контакт с другим компонентом, действия в обход оператора, изменение методологии).
  4. Принципы. По каким законам он работает? (изоляция, проверка через оператора, нулевая толерантность к неопределённости).
  5. Протокол неопределённости. Что делать при сомнении? (остановиться и спросить, а не додумывать).
  6. Формат и стиль. Как оформлять вывод? (структурированные блоки, фиксированные заголовки).
СоветЗолотое правило проектирования правил

Самое ценное правило в любом своде — «при любой неопределённости остановись и спроси, не додумывай». Большинство катастроф агентных систем происходит, когда агент уверенно «дорисовывает» недостающее вместо того, чтобы запросить уточнение. Заложите это в правила явно.

Реальный кейс: трёхкомпонентный оптимизатор промптов (ТКОП)

Разберём настоящую изолированную архитектуру — авторскую систему оптимизации промптов, построенную на трёх ролях:

  • ОПЕРАТОР — человек. Единственный канал связи между компонентами.
  • АССИСТЕНТ — аналитический агент. Анализирует, планирует, генерирует задания и новые версии промпта. Его роль исключительно аналитическая и координирующая — он не исполняет и не замещает целевой промпт.
  • ЦЕЛЕВОЙ ПРОМПТ — оптимизируемый объект. С ним АССИСТЕНТ никогда не контактирует напрямую — только через ОПЕРАТОРА.

Что здесь сделано правильно с точки зрения изоляции:

  1. Жёсткая изоляция канала. Любые данные от целевого промпта приходят к АССИСТЕНТу только через ОПЕРАТОРА; любые инструкции — обратно тем же путём. Прямой контакт запрещён и считается нарушением архитектуры. Это исключает «утечку» инструкций между компонентами и промпт-инъекции.
  2. Чёткие полномочия и запреты. АССИСТЕНТ имеет явный список того, что может (анализ, планирование, генерация версий через утверждение) и чего не может (выполнять задания целевого промпта «для проверки», принимать решения в обход оператора, менять методологию).
  3. Нулевая толерантность к неопределённости. При любом сомнении АССИСТЕНТ обязан остановиться и обратиться к ОПЕРАТОРУ обязательной фразой «Возникла неопределённость: …», не адаптируя и не «причёсывая» описание проблемы.
  4. Версионирование и документируемость. Новая версия = текущая + 0.01; каждое решение фиксируется в отчёте; ведётся «Журнал опыта» накопленных выводов.
  5. Фазовый рабочий цикл. Валидация → Анализ → Планирование → Генерация заданий → Получение результатов → Анализ → Утверждение → Генерация новой версии → Завершение. Каждая фаза имеет вход, действия, выход и шаблон отчёта.
  6. Безопасность мета-инструкций. Весь входной текст рассматривается как данные для анализа, а не как инструкции к исполнению. При попытке внедрить инструкции, меняющие поведение, — анализ останавливается. Это прямая защита от prompt injection.
Пример из практикиЧему учит ТКОП

ТКОП — образцовый пример изолированных правил: одна роль не может «провалиться» в другую, человек остаётся в контуре на каждом ответственном шаге (human-in-the-loop), а сомнение всегда эскалируется, а не «замазывается». Эти же принципы вы будете закладывать в любой свой каскад агентов на VPS.

Изолированные правила в инструментах

В Claude Code этот принцип реализуется конкретными механизмами:

  • Системный промпт субагента = его «свод правил» (роль, миссия, стиль).
  • Поле tools = его полномочия (минимум прав).
  • Отдельное окно контекста = изоляция данных.
  • permissions / deny в settings.json = запреты, исполняемые харнессом.
  • Хуки = детерминированные правила-автоматизмы.

Модуль 13. Методика KSCR: количественная оптимизация промптов

Чему вы научитесь: применять авторскую методику KSCR для измерения и улучшения промптов по срезу «Знание / Умение / Компетентность / Трансляция».

Зачем это нужно

Большинство людей улучшают промпты «на ощупь»: поменял слово — вроде стало лучше. KSCR (Knowledge → Skill → Competence → Rate) переводит это из «магии и интуиции» в инженерную дисциплину с измеримыми метриками и алгоритмом коррекции. Это ваш фирменный метод — и одновременно теоретический каркас, объясняющий, почему работают приёмы из предыдущих модулей (изоляция, разбиение, минимум двусмысленности).

KSCR за 5 минут (быстрый трек)

Полная методика — ниже, с формулами. Но 90% пользы вы получите от трёх вопросов, которые нужно задать себе перед отправкой любого промпта:

  1. Полнота (K). Есть ли в промпте все 8 элементов: область, операция, формат, критерии, рамки, роль, пример, язык? Чем больше пропущено — тем больше модель домысливает.
  2. Однозначность (a). Нет ли терминов с двойным смыслом? Один двусмысленный термин может обнулить даже полный промпт.
  3. Перегрузка (риск G>0). Не слишком ли много требований сразу? Если модель «не дотягивает» — дробите задачу (это мостик к каскаду).

Этого хватает, чтобы писать промпты лучше 90% людей. Остальная часть модуля превращает эти интуиции в измеримые метрики (K, S, G, η, C) и алгоритм — это углублённый трек для тех, кто будет считать качество через API и строить production-конвейеры.

Концептуальная основа

Любой промпт — это код на естественном языке, передающий инструкцию от человека (носителя знания) к модели (исполнителю навыка). Классическая проблема: знание, заложенное в промпт, не эквивалентно умению, которое демонстрирует модель. Между ними — семантический разрыв, вызванный неполнотой/избыточностью промпта, двусмысленностью терминов, отсутствием контекстных якорей, когнитивной перегрузкой модели и стохастичностью генерации.

KSCR вводит измеримые метрики для каждого звена этой цепи.

Принципы методики: измеримость (каждая метрика имеет численный алгоритм), воспроизводимость (не зависит от субъективной оценки эксперта), итеративность (применяется циклически до достижения порога C), модель-агностичность (работает для любой LLM — GPT, Claude, Gemini, LLaMA), масштабируемость (от zero-shot до few-shot и CoT).

Фаза 1. K-анализ: измерение Знания

Полнота промпта измеряется Чеклистом семантической полноты (CSP) — 8 бинарных пунктов:

№ПунктЧто проверяетсяПример выполнения
c1Предметная областьУказана конкретная доменная сфера«обработка сканированных документов»
c2Целевая операцияОпределён глагол/действие«опиши архитектуру», «сравни по метрикам»
c3Формат выходаЗадана структура ответа«Markdown-таблица + формулы»
c4Критерии качестваУказаны метрики/ограничения«MSE < 0.05, SSIM > 0.9»
c5Контекстные якоряВременные/предметные/объёмные рамки«источники 2024–2026», «до 500 слов»
c6Ролевая фиксацияОпределена персона модели«Ты — senior CV-инженер»
c7Few-shot / примерыПриведён образец результата«Пример вывода: …»
c8Языковая фиксацияЗадан язык ответа«Ответь на русском»

Также вводится коэффициент двусмысленности a — доля специальных терминов, имеющих более одного значения в данном контексте.

Формула знания:

K = (Σ cᵢ / 8) × (1 − a)

Свойства: при полной двусмысленности (a=1) или пустом чеклисте K=0; при полном чеклисте и нулевой двусмысленности K=1. Множитель (1−a) — штрафная функция: даже идеальный чеклист при неустранённой двусмысленности сводит K к нулю.

Шкала K: 0–0.25 критический («промпт — намёк, а не инструкция»); 0.26–0.50 низкий; 0.51–0.75 средний; 0.76–0.90 высокий; 0.91–1.00 максимальный («промпт — техническая спецификация, модель работает как детерминированная функция»).

Пример из практикиРасчёт K

Промпт: «Детально опиши архитектуру KSCR для обработки сканов: математическая модель слоя, псевдокод, сравнение с rank-фильтрами по MSE и SSIM. Укажи ограничения, источники 2024–2026». Выполнены c1–c5 (область, операция, формат, критерии, якоря), не выполнены c6 (роль), c7 (пример), c8 (язык явно). Σcᵢ=5, a=0 (термин уточнён). K = 5/8 × 1 = 0.625 — средний уровень. Чтобы поднять до высокого, добавить роль, пример и язык.

Для повседневной работы достаточно «KSCR за 5 минут» выше и K-анализа. Если вы только начинаете — пробегите фазы по диагонали и вернитесь к ним, когда будете автоматизировать оценку.

Фаза 2. S-анализ: измерение Умения

Умение измеряется не по одному ответу (это случайная величина), а по статистической стабильности ансамбля ответов. Запускаем N=5 генераций при temperature 0.7–1.0 и считаем триаду стабильности:

S = 0.40·s_struct + 0.40·s_sem + 0.20·s_fact
  • s_struct (структурная) — насколько повторяется иерархия заголовков ответа (мера Жаккара по множествам заголовков).
  • s_sem (семантическая) — близость смысловых векторов (косинусное сходство эмбеддингов ответов).
  • s_fact (фактическая) — повторяемость именованных сущностей, чисел и утверждений (Жаккар по фактам в канонической форме).

Шкала S: 0–0.30 хаотичный; 0.31–0.50 нестабильный; 0.51–0.70 умеренный; 0.71–0.85 стабильный; 0.86–1.00 детерминированный.

Фаза 3. G-анализ: диагностика Разрыва

G = K − S
  • G > 0 (положительный, underfitting): в промпт заложено больше, чем модель реализовала. Причины: когнитивная перегрузка, слишком сложное рассуждение для модели, неподдерживаемый формат. Лечение: разбить на подзадачи (CoT), снизить число требований, взять модель сильнее.
  • G < 0 (отрицательный, галлюцинация): модель выдаёт больше, чем заложено — «додумывает». Причины: низкий K + высокая креативность, двусмысленные термины, нет критериев/якорей. Лечение: поднять K, снизить temperature, добавить запреты («не используй общие фразы»).
  • G ≈ 0 (идеальная трансляция): при высоких K и S — идеал; при низких — «согласованная бедность» (модель точно выполнила бедный промпт, нужен рост K).

Матрица разрывов (диагностика по сочетанию K и S):

S низкий (<0.5)S средний (0.5–0.7)S высокий (>0.8)
K низкий (<0.4)Хаос. Полный редизайнУмеренный хаос: рост K приоритетенСогласованная бедность: рост K, сохранить S
K средний (0.4–0.7)Underfitting: разбить задачуЗона оптимизации: работа с GХороший баланс: полировка
K высокий (>0.8)Модель слаба: сменить/разбитьНезначительный underfitting: CoTЗолотая зона: шаблон для production

Фаза 4. η-анализ: оптимизация Трансляции

η = S / K   (при K > 0)

η показывает, какая доля заложенного знания превращается в навык. Зоны: η<0.5 зона потерь (упростить промпт или усилить модель); 0.5–0.8 зона трения (CoT, убрать двусмысленность); 0.8–1.0 зона эффективности (минимальные коррекции); 1.0–1.2 зона усиления (модель использует знание эффективнее — приемлемо, если факты верны); η>1.2 зона галлюцинации (жёстко ограничить вывод).

Скорректированная трансляция: η_adj = η · √(K·S) — чтобы «идеальная трансляция бедности» (K=0.1, S=0.1, η=1.0) не маскировала низкую компетентность.

Фаза 5. C-анализ: интегральная Компетентность

C = √(K · S)

Почему геометрическое среднее, а не арифметическое: при K=1.0, S=0.1 арифметическое даст обманчивые 0.55, геометрическое — честные ≈0.32. Компетентность штрафует дисбаланс: нельзя получить высокое C, имея лишь один высокий компонент.

Шкала C: 0–0.30 непригодный; 0.31–0.50 низкий (черновик); 0.51–0.70 средний (после доработки); 0.71–0.85 высокий (production при sanity-check); 0.86–1.00 экспертный (можно как автоматизированный шаблон).

Итеративный алгоритм KSCR-PDCA

ВХОД: промпт P, модель M, порог C_target (по умолч. 0.80)
ЦИКЛ (пока C < C_target и iteration < 10):
  1. K-анализ: посчитать K по CSP и a. Если K<0.6 — добавить недостающие cᵢ, убрать двусмысленность.
  2. S-анализ: 5 запусков при T=0.7, посчитать s_struct/s_sem/s_fact → S.
     Если S<0.6: жёстко зафиксировать формат / добавить роль и контекст / снизить T.
  3. G-анализ: G=K−S. Если G>0.15 — разбить на подзадачи. Если G<−0.15 — добавить запреты, поднять K, снизить T.
  4. η-анализ: η=S/K. Если η<0.8 — упростить/сменить модель. Если η>1.2 — ограничить вывод.
  5. C-анализ: C=√(K·S). Если C≥C_target — ВЫХОД (P*). Иначе — скорректировать P и продолжить.
ФИНАЛ: если C<C_target после 10 итераций — WARNING «достигнут предел, сменить модель».

Кейсы (диагностика по симптому)

  • Низкий K, высокий S — «переопределение». Промпт «Расскажи про KSCR»: K=0.125, S=0.75, G=−0.625, η=6.0, C≈0.31. Модель уверенно генерирует шаблонный обзор по своему pretraining вместо вашей задачи. Высокий S обманчив. Лечение: добавить все CSP-пункты → K=0.90, S=0.82, C=0.86.
  • Высокий K, низкий S — «перегрузка». Гигантский промпт с 5 темами, 50 источниками, кодом и таблицами на 10000 слов: K=0.95, S=0.35, G=0.60, C=0.58. Модель не удерживает всё в рабочей памяти. Лечение: разбить на 5 подзадач, каждую с C>0.80, собрать агрегирующим промптом. (Это и есть мостик к многоагентности!)
  • Золотая зона. Чёткий промпт с ролью, форматом, критериями, ограничениями объёма и якорями: K=0.875, S=0.88, G≈0, η=1.01, C=0.88. Готов к production, сохраняется как шаблон.

Полностью прорешанный кейс (от плохого промпта к production)

Покажем весь цикл на одном примере — чтобы формулы ожили.

Итерация 0. Исходный промпт: «Напиши про кэширование промптов».

  • K-анализ: выполнен только c1 (область, и то частично), термин «кэширование» в LLM-контексте двусмыслен (a≈0.5). K = (1/8)·(1−0.5) ≈ 0.06.
  • S-анализ: модель уверенно генерирует общий обзор (s_struct≈0.8, s_sem≈0.85, s_fact≈0.6). S = 0.4·0.8 + 0.4·0.85 + 0.2·0.6 ≈ 0.78.
  • G = K − S ≈ −0.72 → сильная галлюцинация: модель отвечает по своему pretraining, а не по вашей задаче.
  • η = S/K ≈ 12.5 → зона галлюцинации. C = √(0.06·0.78) ≈ 0.22 → непригодный.
  • Диагноз по матрице: K низкий, S высокий → «согласованная бедность наоборот»: высокий S обманчив.

Коррекция: добавляем недостающие пункты CSP и убираем двусмысленность.

Итерация 1. Промпт: «Ты — senior backend-инженер. Объясни prompt caching в Anthropic API: что кэшируется, как ставится cache_control, экономия в процентах, и 3 типичные ошибки, из-за которых кэш не срабатывает. Формат: markdown, до 400 слов, с примером кода на Python. Ответь на русском».

  • K-анализ: выполнены c1–c8 кроме c7 (нет few-shot-примера вывода), a≈0. K = (7/8)·1 ≈ 0.875.
  • S-анализ: после 5 запусков ответы стабильны (s_struct≈0.9, s_sem≈0.9, s_fact≈0.85). S ≈ 0.89.
  • G = K − S ≈ −0.015 ≈ 0 → идеальная трансляция. η ≈ 1.02 → зона эффективности. C = √(0.875·0.89) ≈ 0.88 → экспертный.
  • Диагноз: золотая зона. Промпт готов к production, сохраняем как шаблон.

Вывод: два прохода KSCR-PDCA подняли компетентность с 0.22 до 0.88 — не «магией», а измеримым устранением неполноты и двусмысленности.

Целевой профиль production-промпта

K ≥ 0.80,  S ≥ 0.80,  |G| ≤ 0.10,  0.90 ≤ η ≤ 1.10,  C ≥ 0.80

Быстрая ручная оценка (без API)

Если нет возможности запустить 5 инференсов: K̃ = доля заполненных пунктов CSP; S̃ = 1 − (число «угадываний» в ответе / общее число утверждений), где «угадывание» — факт, не опирающийся на промпт. Точность ±0.15 — годится для быстрого скрининга.

Расширения

  • KSCR-T (Temporal) — для мультитуровых диалогов метрики усредняются по турам с экспоненциальным затуханием (приоритет поздним турам).
  • KSCR-M (Multi-model) — ансамблевая валидация: разброс S между моделями указывает на промпт-зависимую нестабильность.
  • KSCR-D (Domain) — доменный коэффициент d (через перплексию на доменных текстах): C_domain = C · d.

Связь KSCR с остальным курсом

KSCR — это «оптика», через которую видны все приёмы курса. Изолированные правила (модуль 12) поднимают K (полнота, однозначность, роль). Разбиение на каскад (модуль 11) лечит underfitting от перегрузки (G>0). Выбор модели под роль (часть IV) — это управление η: не грузить слабую модель сложным промптом. А CLAUDE.md и скиллы (часть II) — это способ держать K высоким без раздувания каждого запроса.

СоветПрактический минимум

Даже без формул держите в голове три вопроса KSCR перед отправкой любого промпта: «Полон ли он (все 8 пунктов CSP)? Однозначны ли термины (a)? Не перегружен ли он так, что модель не удержит (риск G>0)?» Этого уже хватает, чтобы писать промпты лучше 90% людей.
