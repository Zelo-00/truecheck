ЧАСТЬ VI. АГЕНТЫ ВНЕ CLI: СТРОИМ ПРОДУКТ

До сих пор агент жил внутри Claude Code (терминал) или n8n (визуальный редактор). Но настоящая цель AI-инженера — агент, который работает сам, как сервис: принимает события, крутится по расписанию, отвечает в Telegram, обрабатывает очередь — без вашего присутствия. В этой части мы построим такого агента кодом и развернём на VPS, а затем соберём из нескольких агентов автономный каскад.

Модуль 20. Claude Agent SDK и Anthropic SDK: три способа построить агента

Чему вы научитесь: выбирать уровень абстракции для своего агента и писать агентный цикл на коде.

Зачем это нужно

«Построить агента» можно на разной высоте: от полного контроля над циклом до готового хостинга. Понимание трёх уровней позволяет не «изобретать велосипед» и не «тащить комбайн» там, где хватит функции.

Три поверхности (surfaces)

УровеньЧто этоКогда брать
Один вызов APIЗапрос → ответ (классификация, суммаризация, извлечение)Простая задача без многошаговости
Workflow (tool use)Вы сами управляете циклом «вызов → инструмент → результат»Многошаговые пайплайны с вашими инструментами
Агент (Agent SDK / Managed Agents)Модель сама решает траекторию; цикл ведёт SDK или AnthropicОткрытые, плохо формализуемые задачи
СоветНачинайте с простого

Не каждая задача требует агента. Если хватает одного вызова или workflow — берите их: дешевле, быстрее, надёжнее. Агент — для задач, где модель должна сама исследовать и принимать решения (вспомните 4 критерия из модуля 10).

Способ 1. Claude Agent SDK — каркас Claude Code в вашем коде

Claude Agent SDK — это тот же агентный движок, что и в Claude Code, но как библиотека для встраивания в ваше приложение (Python и TypeScript). Он даёт готовый агентный цикл с инструментами файловой системы, bash, поддержкой MCP, субагентов, хуков и скиллов — всё, что вы освоили в части II, но программно.

Идея использования (Python, концептуально):

# pip install claude-agent-sdk
from claude_agent_sdk import query, ClaudeAgentOptions

options = ClaudeAgentOptions(
    system_prompt="Ты — агент, который наводит порядок в каталоге логов: "
                  "анализирует .log файлы и пишет сводку в summary.md.",
    allowed_tools=["Read", "Write", "Bash", "Grep"],   # минимум прав
    # mcp_servers=[...], permission_mode=..., и т.д.
)

async for message in query(prompt="Сделай сводку по логам за сегодня", options=options):
    print(message)

Здесь Claude Code «помогает строить агента, работающего вне CLI» в прямом смысле: вы используете его же возможности (скиллы, субагенты, MCP, права) из своего сервиса. Точные сигнатуры и опции смотрите в официальной документации Agent SDK — API развивается.

ЗаметкаКак Claude Code помогает строить таких агентов

Двояко. Во-первых, Claude Agent SDK — это и есть «движок Claude Code как библиотека». Во-вторых, сам Claude Code — отличный инструмент, чтобы написать код вашего агента: спроектировать, реализовать, протестировать. Вы строите агента с помощью агента.

Способ 2. Anthropic SDK — полный контроль над циклом

Когда нужен максимальный контроль (свои инструменты, проверки, human-in-the-loop, логирование каждого шага), используют прямой Anthropic SDK с tool use. Два варианта: автоматический «tool runner» и ручной цикл.

Автоматический tool runner (Python) — SDK сам ведёт цикл:

import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()   # ключ из ANTHROPIC_API_KEY (в .env)

@beta_tool
def get_disk_usage(path: str) -> str:
    """Вернуть использование диска по пути.
    Args:
        path: путь к каталогу.
    """
    import shutil
    total, used, free = shutil.disk_usage(path)
    return f"used={used//2**30}GB free={free//2**30}GB"

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=16000,
    tools=[get_disk_usage],
    messages=[{"role": "user", "content": "Сколько занято на /var?"}],
)
for message in runner:
    print(message)

Ручной агентный цикл (Python) — когда нужно вклиниваться между шагами:

messages = [{"role": "user", "content": user_input}]
while True:
    response = client.messages.create(
        model="claude-opus-4-8", max_tokens=16000, tools=tools, messages=messages,
    )
    if response.stop_reason == "end_turn":
        break
    messages.append({"role": "assistant", "content": response.content})
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)   # ваша реализация
            tool_results.append({
                "type": "tool_result", "tool_use_id": block.id, "content": result,
            })
    messages.append({"role": "user", "content": tool_results})
ВажноДешёвый каскад вне CLI — через OpenRouter

Anthropic SDK работает с моделями Claude. Если вы строите дешёвый каскад на open-моделях, используйте OpenAI-совместимый клиент, указывающий на OpenRouter (часть IV): тот же агентный цикл, но client = OpenAI(base_url="https://openrouter.ai/api/v1", ...) и модель вроде deepseek/deepseek-v3.2. Сильные шаги — на Claude через Anthropic SDK, массовые — на дешёвой модели через OpenRouter. Это и есть каскад в коде.

Способ 3. Managed Agents — Anthropic ведёт цикл и держит песочницу

Managed Agents — серверная поверхность: вы создаёте сохранённую конфигурацию агента (модель, системный промпт, инструменты, MCP, скиллы), а Anthropic запускает сессии, провижинит изолированный контейнер для исполнения инструментов и стримит события. Подходит, когда не хотите держать собственный рантайм исполнения инструментов и нужен надёжный stateful-агент с рабочим пространством.

Обязательный порядок: сначала агент (один раз), потом сессии (на каждый запуск).

# 1. Создать агента ОДИН раз, сохранить agent.id
agent = client.beta.agents.create(
    name="Log Analyst",
    model="claude-opus-4-8",
    system="Ты анализируешь логи и пишешь сводки.",
    tools=[{"type": "agent_toolset_20260401"}],
)
# 2. На каждый запуск — сессия, ссылающаяся на agent.id
session = client.beta.sessions.create(
    agent=agent.id, environment_id=environment.id,
)
ВниманиеАнтипаттерн Managed Agents

Не вызывайте agents.create() на каждый запуск — это плодит «осиротевших» агентов и платит лишнюю задержку. Создайте агента один раз (в скрипте настройки), сохраните ID, и на каждый запуск создавайте только сессию.

Чеклист модуля 20

  • Понимаю три поверхности (вызов / workflow / агент) и когда какую брать.
  • Знаю, что Claude Agent SDK = движок Claude Code как библиотека.
  • Умею написать агентный цикл на Anthropic SDK (tool runner или ручной).
  • Понимаю, как сделать дешёвый каскад через OpenRouter.
  • Знаю порядок Managed Agents: агент один раз → сессии на каждый запуск.

Упражнения

  1. Напишите простой workflow-агент на Anthropic SDK с одним своим инструментом (например, «прочитать погоду по городу» — заглушкой). Признак успеха: в логе видно, что агент вызвал ваш инструмент (tool_use) и использовал его результат в ответе.
  2. Перепишите тот же цикл на OpenAI-совместимый клиент к OpenRouter с дешёвой моделью.

Модуль 21. Деплой агента как сервиса на VPS

Чему вы научитесь: превращать скрипт-агент в постоянно работающий сервис: systemd/Docker, cron, вебхук, Telegram-бот как фронт.

Зачем это нужно

Скрипт, запущенный руками в терминале, умрёт, когда вы закроете SSH. Сервис работает всегда и сам перезапускается при сбое. Это финальный шаг «агент → продукт».

Способ A. systemd-сервис

Положите агента в /home/deploy/agent/app.py (виртуальное окружение, зависимости установлены). Создайте unit-файл /etc/systemd/system/myagent.service:

[Unit]
Description=My AI Agent
After=network.target

[Service]
Type=simple
User=deploy
WorkingDirectory=/home/deploy/agent
EnvironmentFile=/home/deploy/agent/.env
ExecStart=/home/deploy/agent/.venv/bin/python app.py
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target

Активируйте:

sudo systemctl daemon-reload
sudo systemctl enable --now myagent
sudo systemctl status myagent          # проверить
journalctl -u myagent -f               # смотреть логи

Теперь агент работает постоянно, переживает перезагрузку сервера и сам перезапускается при падении (Restart=on-failure).

Способ B. Docker-контейнер

Заверните агента в образ и запустите так же, как n8n. Преимущество — изоляция и переносимость. Минимальный Dockerfile:

FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["python", "app.py"]

Запуск: docker build -t myagent . && docker run -d --restart unless-stopped --env-file .env --name myagent myagent.

Способ C. По расписанию (cron)

Если агент должен срабатывать периодически (раз в час проверить почту), не держите процесс постоянно — запускайте по cron:

crontab -e
# каждый час в :00 запускать агента
0 * * * * /home/deploy/agent/.venv/bin/python /home/deploy/agent/check.py >> /home/deploy/agent/logs/cron.log 2>&1

Способ D. Вебхук-фронт

Чтобы внешний мир (другой сервис, форма, GitHub) триггерил агента, поднимите лёгкий веб-сервер (FastAPI) с эндпоинтом, который при POST запускает агентный цикл:

from fastapi import FastAPI, Request
app = FastAPI()

@app.post("/run")
async def run(req: Request):
    data = await req.json()
    result = run_agent(data["task"])     # ваш агентный цикл
    return {"result": result}

Запускайте через uvicorn под systemd, закройте за обратным прокси с HTTPS и проверкой токена.

Способ E. Telegram-бот как фронт (самый удобный)

Telegram-бот — отличный «фронт» для личного агента: пишете боту в мессенджер, он выполняет агентный цикл и отвечает. Каркас на python-telegram-bot:

from telegram.ext import Application, MessageHandler, filters

async def on_message(update, context):
    user_text = update.message.text
    answer = run_agent(user_text)         # ваш агентный цикл (Anthropic/OpenRouter)
    await update.message.reply_text(answer)

app = Application.builder().token(TELEGRAM_BOT_TOKEN).build()
app.add_handler(MessageHandler(filters.TEXT & ~filters.COMMAND, on_message))
app.run_polling()

Токен бота берётся у @BotFather, кладётся в .env, сервис — под systemd. Так у вас в кармане появляется личный агент, доступный из мессенджера 24/7.

ВниманиеБезопасность бота

Ограничьте, кто может писать боту (проверка update.effective_user.id по белому списку), иначе любой найдёт его и будет тратить ваши токены. И никогда не кладите токен бота в код или публичный репозиторий.

Пример из практикиНаш сквозной проект

Теперь у «сортировщика входящих» есть постоянный фронт: вебхук (или Telegram) принимает сообщения 24/7, а сервис под systemd сам перезапускается при сбоях. Осталось собрать полный каскад с ролями и судьёй — это модуль 22 и капстоун.

Чеклист модуля 21

  • Знаю Docker-вариант деплоя.
  • Умею настроить cron для периодического агента.
  • Поднял вебхук- или Telegram-фронт для агента.
  • Ограничил доступ и защитил секреты.

Упражнения

  1. Разверните своего агента из модуля 20 как systemd-сервис и убедитесь, что он переживает reboot.
  2. Сделайте Telegram-бота-фронт с белым списком пользователей.

Модуль 22. Каскад автономных агентов на вашем VPS

Чему вы научитесь: проектировать и разворачивать каскад из нескольких автономных агентов на одном сервере, следить за ним и считать стоимость.

Зачем это нужно

Это кульминация курса: всё вместе. Каскад автономных агентов на VPS — это система, которая принимает события, распределяет работу между специализированными агентами на разных (преимущественно дешёвых) моделях, исполняется надёжно и стоит копейки. Здесь сходятся инфраструктура (часть I), оркестрация (часть III), экономика моделей (часть IV) и деплой (этот модуль).

Эталонная архитектура каскада на VPS

            ВНЕШНИЙ МИР (почта, Telegram, вебхуки, расписание)
                              │
                       [ Шлюз / очередь ]      ← FastAPI вебхук или n8n + Redis
                              │
                   ┌──────────┴──────────┐
                   │   АГЕНТ-РОУТЕР       │      ← дешёвая модель (Mistral Nemo)
                   │  классификация       │        «куда направить?»
                   └──────────┬──────────┘
              ┌───────────────┼────────────────┐
              ▼               ▼                ▼
        [ВОРКЕР-A]       [ВОРКЕР-B]       [ВОРКЕР-C]    ← дешёвые модели (DeepSeek Flash,
        извлечение       RAG по базе      эскалация       Mistral Small, Llama Scout)
              └───────────────┼────────────────┘
                              ▼
                   ┌─────────────────────┐
                   │  АГЕНТ-СУДЬЯ/СБОРКА   │      ← аккуратная модель (Claude Haiku / GPT-5 Mini)
                   │  проверка + ответ     │        эскалация на сильную при провале
                   └──────────┬──────────┘
                              ▼
                     ОТВЕТ / ДЕЙСТВИЕ (БД, API, сообщение)

Принципы, которые вы уже знаете, в сборе:

  • Изоляция (часть III): у каждого агента — свой системный промпт (свод правил), свои инструменты (минимум прав), свой контекст. Роутер не умеет писать в БД; воркер не принимает решений об эскалации.
  • Каскад/экономика (часть IV): 90% объёма — на дешёвых моделях; сильная модель подключается только когда судья забраковал результат.
  • Передача контекста (часть III): агенты передают друг другу цель и сжатый результат, а не «сырые дампы».
  • KSCR (часть III): каждый агент-роль имеет узкий, полный (высокий K), однозначный промпт — поэтому η высок, а C ≥ 0.8.

Чем оркестрировать

Два практичных пути на одном VPS:

  1. n8n как оркестратор (часть V): супервизор + под-воркфлоу, очередь на Redis. Меньше кода, визуально, легко менять.
  2. Свой код-оркестратор (этот раздел): FastAPI-шлюз кладёт задачи в очередь (Redis/БД), воркеры-сервисы под systemd забирают и исполняют агентные циклы. Больше контроля, лучше для сложной логики.

Часто берут гибрид: n8n для интеграций и простых веток, свой код — для тяжёлой агентной логики, связанные через вебхуки.

Пример из практикиМинимальный автономный каскад на VPS

На сервере крутятся: Postgres (память и очередь), Redis (очередь n8n/воркеров), n8n (роутер + интеграции), 2–3 Python-воркера под systemd (специалисты), Telegram-бот (фронт). Всё в Docker/systemd, секреты в .env с chmod 600, файрвол закрыт, бэкап БД по cron. Стоимость инференса — единицы долларов в месяц при тысячах обработанных событий, потому что тяжёлая модель почти не вызывается.

Сколько это стоит: расчёт на 1000 событий

Бриф курса требует уметь считать стоимость. Покажем на числах. Формула простая:

стоимость_роли = (вх_токены/1e6 × цена_вх) + (вых_токены/1e6 × цена_вых)
стоимость_события = Σ_по_ролям (стоимость_роли × доля_вызова роли)

Возьмём каскад «сортировщик входящих», 1000 событий. Цены — приблизительные, $/1М токенов (часть IV):

РоутерMistral Nemo (0.02/0.03)400 / 50100%≈ $0.01
ВоркерDeepSeek V4 Flash (0.09/0.18)1500 / 600100%≈ $0.24
СудьяClaude Haiku (1/5)1000 / 300100%≈ $2.50
ЭскалацияOpus (5/25)3000 / 15008%≈ $4.20

Пример из практикиКак раздвинуть экономию ещё сильнее

В этом расчёте дороже всего — судья (Haiku) и эскалация. Заменив судью на более дешёвую модель (GPT-5 Mini 0.25/2 или DeepSeek V3.2 0.23/0.34) и снизив долю эскалации до 3–5% за счёт хорошего роутера, тот же каскад выходит на 15–20× дешевле «всё на Opus». Плюс prompt caching на неизменном системном промпте срезает ещё 60–80% входных токенов. Вот откуда берётся «на порядок дешевле».

Главный вывод: считайте по этой формуле до запуска, и вы увидите, какая роль «съедает» бюджет. Обычно это судья и эскалация — туда и направляйте оптимизацию.

Какие агенты и модели лучше использовать

  • Роутер: самая дешёвая надёжная модель (Mistral Nemo, DeepSeek V4 Flash). Задача простая — классификация.
  • Воркеры: дешёвые рабочие лошадки под тип задачи (DeepSeek V3.2 — универсал; Mistral Small — компактно; Llama Scout — длинный контекст; Qwen3 Coder — код).
  • Суммаризатор: Gemini Flash-Lite (дёшево, длинный вход) или Claude Haiku (аккуратность).
  • Судья/сборка: Claude Haiku или GPT-5 Mini (надёжный формат, следование инструкциям).
  • Эскалация (трудные случаи): сильная фронтир-модель (Claude Opus / DeepSeek V4 Pro) — но только по необходимости.

Мониторинг и надёжность

  • Логи: каждый агент пишет структурированный лог (что получил, что сделал, сколько токенов). journalctl -u <сервис> -f и/или централизованный лог-файл.
  • Здоровье: docker ps, systemctl status, df -h (диск под логами), free -h. Привычка ежедневного взгляда.
  • Стоимость: считайте токены на каждом шаге; у каждого API-ключа — лимит расхода (часть I). Дешёвый «судья» перед дорогой эскалацией экономит больше всего.
  • Устойчивость: Restart=on-failure, fallback-цепочки в OpenRouter, retry с экспоненциальной задержкой на 429/5xx, очередь, которая переживёт перезапуск.
  • Безопасность: изоляция (Docker), минимум прав у каждого агента, human-in-the-loop на необратимых действиях (отправка денег, удаление, публикация).
