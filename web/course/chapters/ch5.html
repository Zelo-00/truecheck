ЧАСТЬ V. N8N — ОРКЕСТРАЦИЯ АГЕНТОВ БЕЗ КОДА

Claude Code и Agent SDK — это про код. Но огромный класс агентных задач (автоматизации, интеграции, обработка событий) удобнее собирать визуально, соединяя блоки мышкой. Для этого есть n8n — мощная open-source платформа автоматизации с полноценной поддержкой ИИ-агентов. В этой части мы развернём n8n на нашем VPS и соберём в нём настоящий каскад агентов.

Модуль 17. Установка n8n и первый workflow

Чему вы научитесь: разворачивать n8n в Docker на VPS, понимать его модель «нод и связей» и собирать первый рабочий процесс.

Зачем это нужно

n8n — это «конструктор автоматизаций»: вы соединяете блоки (ноды) в workflow, где данные текут от одной ноды к другой. Триггер (входящее письмо, вебхук, расписание) запускает цепочку, а внутри неё можно вызывать ИИ-агента. Это идеальный мост между «миром событий» (почта, чаты, API) и «миром LLM».

Теория: ноды, связи, workflow

  • Нода (node) — блок, выполняющий одно действие: HTTP-запрос, чтение из БД, отправка в Telegram, вызов ИИ-агента.
  • Связь — стрелка, по которой данные (в формате JSON) переходят от ноды к ноде.
  • Workflow — весь граф нод. Запускается триггером: Webhook (входящий HTTP), Schedule (по расписанию/cron), Chat (сообщение), вручную.

Установка на VPS через Docker

Создаём том для постоянного хранения и запускаем официальный образ. Базовый вариант:

docker volume create n8n_data

docker run -d --restart unless-stopped \
  --name n8n \
  -p 5678:5678 \
  -e GENERIC_TIMEZONE="Europe/Moscow" \
  -e TZ="Europe/Moscow" \
  -v n8n_data:/home/node/.n8n \
  docker.n8n.io/n8nio/n8n

n8n поднимется на порту 5678. Если вы на сервере — откройте порт в ufw для теста (sudo ufw allow 5678/tcp), затем зайдите на http://<ВАШ_IP>:5678 и создайте аккаунт владельца (логин/пароль) при первом входе.

ВниманиеНе выставляйте n8n голым в интернет надолго

Порт 5678 без HTTPS и за пределами теста — плохая идея: там ваши секреты и доступ к автоматизациям. Для постоянной работы поставьте обратный прокси (Caddy/nginx/Traefik) с HTTPS и доменом, а сам 5678 закройте файрволом. Для вебхуков задайте переменную WEBHOOK_URL=https://n8n.example.com/, иначе внешние вебхуки будут работать некорректно.

Продакшен-вариант: docker compose с PostgreSQL

Для постоянной работы лучше держать n8n с внешней базой PostgreSQL (надёжнее SQLite по умолчанию) через docker-compose.yml:

services:
  postgres:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_USER: n8n
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: n8n
    volumes:
      - pg_data:/var/lib/postgresql/data

  n8n:
    image: docker.n8n.io/n8nio/n8n
    restart: unless-stopped
    ports:
      - "5678:5678"
    environment:
      DB_TYPE: postgresdb
      DB_POSTGRESDB_HOST: postgres
      DB_POSTGRESDB_DATABASE: n8n
      DB_POSTGRESDB_USER: n8n
      DB_POSTGRESDB_PASSWORD: ${POSTGRES_PASSWORD}
      N8N_ENCRYPTION_KEY: ${N8N_ENCRYPTION_KEY}
      GENERIC_TIMEZONE: Europe/Moscow
    volumes:
      - n8n_data:/home/node/.n8n
    depends_on:
      - postgres

volumes:
  pg_data:
  n8n_data:

Поднять: docker compose up -d (переменные POSTGRES_PASSWORD, N8N_ENCRYPTION_KEY — в .env рядом).

ВажноЗафиксируйте N8N_ENCRYPTION_KEY

Этим ключом n8n шифрует ваши учётные данные (credentials). Если потеряете его при пересоздании контейнера — потеряете доступ ко всем сохранённым подключениям. Сгенерируйте раз (openssl rand -hex 16) и храните в .env и в бэкапе.

Первый workflow: «вебхук → ответ»

  1. В интерфейсе создайте новый workflow.
  2. Добавьте ноду-триггер Webhook, задайте путь.
  3. Добавьте ноду Respond to Webhook с текстом ответа.
  4. Соедините их, активируйте workflow и дёрните вебхук (curl).

Поздравляю — вы только что развернули программируемый эндпоинт без единой строки серверного кода.

Пример из практикиНаш сквозной проект

Развёрнутый n8n — это оркестратор нашего «сортировщика входящих». В модуле 19 мы повесим на него Webhook-триггер для приёма сообщений и соберём супервизора с субагентами (роутер → специалисты → ответ).

Модуль 18. ИИ-агенты в n8n: AI Agent node

Чему вы научитесь: собирать ИИ-агента в n8n из узлов «модель + инструменты + память» и подключать OpenRouter или Ollama.

Зачем это нужно

n8n строит ИИ поверх LangChain. Это значит, что прямо в визуальном редакторе вы собираете полноценного агента из блоков: «мозг» (модель), «руки» (инструменты), «память». Получается агент, доступный по вебхуку или чату, без написания кода.

Анатомия AI-узлов

Узлы делятся на корневые (root, к ним всё подключается) и под-узлы (sub-nodes, присоединяются к корневым):

  • Корневой узел: AI Agent. Сердце агента. Варианты: Tools Agent (основной), Conversational, ReAct, Plan-and-Execute, SQL Agent. Рядом — Basic LLM Chain, Question and Answer Chain, Summarization Chain, Text Classifier, Information Extractor.
  • Chat Model (под-узел, «мозг»): OpenAI Chat Model, OpenRouter Chat Model, Ollama Chat Model, Anthropic Chat Model, Google Gemini Chat Model, Groq Chat Model, Mistral Cloud Chat Model и др.
  • Tools (под-узлы, «руки»): Calculator, Custom Code Tool, MCP Client Tool, поисковые инструменты, Vector Store Question Answer Tool, и ключевой для каскада — Call n8n Workflow Tool.
  • Memory (под-узлы, память): Simple Memory, Postgres Chat Memory, Redis Chat Memory, MongoDB Chat Memory.
  • Vector Store (для RAG): Qdrant, Pinecone, Chroma + узлы эмбеддингов.

Схематично агент в n8n собирается так:

        ┌──────────────── AI Agent (root) ────────────────┐
        │                                                 │
        │   ▲ Chat Model (sub)   ── «мозг» (OpenRouter)   │
        │   ▲ Memory (sub)       ── «память» (Postgres)   │
        │   ▲ Tool (sub)         ── Calculator / Code      │
        │   ▲ Tool (sub)         ── Call n8n Workflow Tool │
        └─────────────────────────────────────────────────┘
   (Триггер: Chat / Webhook)  ───▶  AI Agent  ───▶  Ответ / Действие

К корневому узлу AI Agent «снизу» подключаются под-узлы (модель, память, инструменты). Это визуальное воплощение формулы агента из части III: LLM + инструменты + память + цикл.

Сборка первого агента: пошагово

  1. Добавьте триггер Chat (или Webhook).
  2. Добавьте корневой узел AI Agent (тип Tools Agent).
  3. Присоедините к нему под-узел Chat Model → выберите OpenRouter Chat Model и заведите credentials с вашим ключом OpenRouter. В поле модели укажите дешёвую модель, например deepseek/deepseek-v3.2.
  4. Присоедините Memory → Simple Memory (или Postgres для постоянной памяти).
  5. Присоедините нужные Tools (например, Calculator и Custom Code Tool).
  6. Активируйте и поговорите с агентом через чат.
СоветOpenRouter в n8n

Если в вашей версии n8n есть отдельный узел OpenRouter Chat Model — используйте его. Если нет — возьмите OpenAI Chat Model и в его credentials пропишите base URL https://openrouter.ai/api/v1 и ключ OpenRouter. Это тот же приём «OpenAI-совместимости», что мы видели в части IV. Для Ollama — узел Ollama Chat Model с base URL http://host.docker.internal:11434. Важная грабля: n8n у нас в Docker, поэтому localhost внутри контейнера НЕ достучится до Ollama на хосте. На Linux-сервере запустите контейнер с --add-host=host.docker.internal:host-gateway (в compose — extra_hosts: ["host.docker.internal:host-gateway"]), иначе контейнер не увидит Ollama.

Память агента

  • Simple Memory — в оперативке, живёт пока работает процесс. Для теста.
  • Postgres/Redis Chat Memory — постоянная память диалога, переживает перезапуски. Для прода. (У вас уже есть Postgres из compose-файла — переиспользуйте.)

RAG: агент, который знает ваши документы

Чтобы агент отвечал по вашим данным (документация, база знаний), добавляют Vector Store (Qdrant/Chroma) и узлы эмбеддингов: документы режутся на куски, превращаются в векторы и кладутся в хранилище; на запрос агент достаёт релевантные куски и отвечает по ним. Это классический паттерн Retrieval-Augmented Generation.

Модуль 19. Каскад агентов в n8n: supervisor и sub-workflows

Чему вы научитесь: строить многоагентные сценарии в n8n через под-воркфлоу, паттерн «супервизор» и масштабировать систему в queue mode.

Зачем это нужно

Один агент в n8n — это хорошо. Но настоящая мощь — в каскаде: главный агент-супервизор делегирует подзадачи специализированным субагентам. В n8n это собирается из готовых блоков, без кода — прямое воплощение теории из части III.

Ключевой механизм: «Call n8n Workflow Tool»

В n8n один workflow может вызывать другой как инструмент агента. Узел Call n8n Workflow Tool превращает целый под-воркфлоу в инструмент, который главный AI Agent вызывает по необходимости. Это и есть техническая основа паттерна «супервизор».

Паттерн «супервизор» (supervisor)

Архитектура:

  1. Главный агент (супервизор) на сильной модели (например, DeepSeek V3.2 или Claude). Его инструменты — это под-воркфлоу-агенты.
  2. Субагенты-специалисты, каждый — отдельный workflow со своей моделью, инструментами и памятью:
  • «Поисковик» (дешёвая модель + веб-поиск);
  • «Аналитик данных» (модель + SQL/доступ к БД);
  • «Писатель» (модель + шаблоны).
  1. Супервизор получает задачу, решает, кого из специалистов позвать, передаёт им подзадачи (через Call n8n Workflow Tool), собирает результаты и формирует итог.

Это ровно та оркестрация, что мы изучали: разделение труда, изоляция (у каждого субагента свой контекст и права), экономия (специалисты на дешёвых моделях).

Пример из практикиКаскад «обработка обращений» в n8n

Webhook принимает обращение → агент-роутер (Mistral Nemo, дёшево) классифицирует: «жалоба / вопрос / спам» → супервизор по типу зовёт нужного субагента (для вопроса — RAG-агент по базе знаний; для жалобы — агент эскалации, создающий тикет через MCP) → агент-писатель формирует ответ → Respond to Webhook. 90% объёма — на копеечных моделях, дорогая подключается только на сложных случаях.

Триггеры и интеграции

  • Под-воркфлоу запускаются через Execute Sub-workflow / Workflow Trigger.
  • Внешние события — через Webhook-узлы (с корректным WEBHOOK_URL).
  • Сотни готовых интеграций (Telegram, Slack, Gmail, Google Sheets, базы) подключают агента к реальному миру.

Масштабирование: queue mode

Когда нагрузка растёт, n8n переводят в queue mode (режим очереди) на Redis:

  • главный процесс кладёт задания в Redis;
  • один или несколько worker-процессов забирают и исполняют их;
  • масштаб — добавлением воркеров.

Включается переменной EXECUTIONS_MODE=queue на главном процессе и на всех воркерах; нужен доступ к Redis и к Postgres. Это превращает вашу n8n из «одной машинки» в горизонтально масштабируемую систему обработки.
