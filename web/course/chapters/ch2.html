ЧАСТЬ II. CLAUDE CODE — РАБОЧАЯ СТАНЦИЯ AI-ИНЖЕНЕРА

Сервер готов. Теперь установим инструмент, который превращает терминал в среду разработки с ИИ-агентом. Claude Code — официальный CLI-агент от Anthropic: он живёт в вашем терминале, читает и пишет файлы, выполняет команды, запускает тесты и сам ведёт многошаговую работу. В этой части мы освоим его не как «чат в консоли», а как профессиональную платформу: скиллы, субагенты, хуки, MCP и инженерия контекста.

Модуль 6. Установка и подключение Claude Code

Чему вы научитесь: устанавливать Claude Code, авторизоваться, делать первый запуск и понимать модель оплаты.

Зачем это нужно

Claude Code — это «руки» модели на вашей машине. В отличие от веб-чата, он видит ваш проект, запускает реальные команды и доводит задачу до конца сам. Это и есть переход от «спросить нейросеть» к «поручить агенту».

Шаг 1. Предусловия — Node.js

Claude Code распространяется как npm-пакет, поэтому нужен Node.js версии 18 или новее (рекомендуется LTS 20/22). Проверьте:

node --version

Если Node не установлен или старый, поставьте через nvm (Node Version Manager) — это лучший способ держать нужную версию без прав root:

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
# перезапустите shell или выполните:
export NVM_DIR="$HOME/.nvm"; . "$NVM_DIR/nvm.sh"
nvm install 22
nvm use 22
ВниманиеГрабли с nvm

nvm активируется только в текущей сессии shell. В каждом новом терминале (и в системных сервисах) node/npm могут «не найтись», пока вы не выполните строку инициализации export NVM_DIR=...; . "$NVM_DIR/nvm.sh". Это частая причина «у меня же стоял node». Решение — прописать инициализацию в ~/.bashrc.

Шаг 2. Установка Claude Code

npm install -g @anthropic-ai/claude-code

Проверьте:

claude --version

Шаг 3. Первый запуск и авторизация

Перейдите в каталог любого проекта и запустите:

cd ~/my-project
claude

При первом запуске Claude Code предложит войти. Команда авторизации внутри интерфейса — /login. Доступны два способа:

  • Аккаунт Claude (подписка Pro/Max) — вход через браузер, оплата по подписке. Удобно для интенсивной личной работы: фиксированная цена.
  • API-ключ Anthropic — оплата по токенам через Console (console.anthropic.com). Гибко, прозрачно по стоимости, подходит для команд и автоматизации.
СоветПодписка vs API

Для ежедневной плотной работы подписка Max часто выгоднее: вы не считаете токены. Для редких задач, скриптов и серверной автоматизации — API-ключ. Их можно держать раздельно: подписку на ноутбуке, API-ключ на сервере.

Шаг 4. Модели и стоимость

Claude Code работает на моделях семейства Claude. На момент курса актуальны:

МодельIDКонтекстЦена вход / выход за 1М токеновКогда

Переключить модель внутри Claude Code можно командой /model. Важный приём экономии: тяжёлую работу ведёт Opus, а массовые простые подзадачи (поиск, чтение, классификация) отдают субагентам на Haiku — об этом в модуле про субагентов.

ЗаметкаКэширование промптов

Claude автоматически кэширует повторяющийся контекст (системный промпт, описания инструментов). Кэш-чтение стоит около 10% от цены входных токенов. Поэтому стабильный, не меняющийся от запроса к запросу контекст (например, CLAUDE.md) — это и быстрее, и дешевле. Не вставляйте в начало промпта меняющиеся данные (дату-время, случайные ID): любое изменение в начале «сбивает» кэш для всего, что идёт после.

Модуль 7. Анатомия Claude Code: контекст, CLAUDE.md, режимы, права

Чему вы научитесь: управлять контекстом проекта через CLAUDE.md, понимать режимы работы и систему разрешений, настраивать settings.json.

Зачем это нужно

Качество работы агента на 80% определяется тем, что он «знает» о вашем проекте и какие правила соблюдает. Это и есть инженерия контекста — навык, который отличает того, у кого «ИИ пишет мусор», от того, у кого «ИИ пишет как senior».

CLAUDE.md — память проекта

CLAUDE.md — это файл в корне проекта, который Claude Code автоматически читает при запуске и держит в контексте. Сюда кладут то, что агент должен знать всегда:

  • стек и архитектуру («монорепо pnpm: apps/web на React+Vite, apps/api на NestJS+Prisma»);
  • команды («тесты: pnpm test; линт: pnpm lint; запуск dev: pnpm dev»);
  • соглашения («коммиты по Conventional Commits; не трогать каталог legacy/»);
  • ограничения среды («целевой браузер Firefox 116; офлайн-сборка»).
Пример из практикиПример из практики

В реальных проектах на этом сервере CLAUDE.md-подобные инструкции фиксируют жёсткие версии под голый Windows Server 2012 R2: Node 18.x (не выше — 20+ не стартует), PostgreSQL 12, конкретный VC++ Redist. Без такой памяти агент «по умолчанию» взял бы свежие версии и сломал офлайн-деплой. Вывод: контекст — это не «приятно иметь», а защита от целого класса ошибок.

Иерархия памяти:

  • Глобальная — ~/.claude/CLAUDE.md: ваши личные предпочтения для всех проектов (стиль ответов, любимые инструменты).
  • Проектная — <проект>/CLAUDE.md: знания о конкретном проекте, коммитится в репозиторий, общая для команды.
  • Локальная — можно держать неразделяемые заметки вне git.

Хорошее правило: если вы объясняете агенту одно и то же во второй раз — это пора записать в CLAUDE.md.

Режимы работы

  • Обычный режим — агент предлагает и выполняет действия, запрашивая разрешение на потенциально опасные (запись файлов, выполнение команд) согласно настройкам.
  • Plan mode (режим планирования) — агент только исследует и составляет план, ничего не меняя. Идеален, чтобы сначала договориться о подходе на сложной задаче, а потом дать «добро» на реализацию. Включается клавишей/командой плана.
  • Auto-accept — агент выполняет действия без подтверждений (быстрее, но применяйте осознанно, в безопасном окружении).

Система разрешений и settings.json

Claude Code не даёт агенту делать что угодно. Поведение настраивается в settings.json (глобальный ~/.claude/settings.json и проектный .claude/settings.json):

  • permissions — что разрешено без спроса, что запрещено. Например, можно разрешить чтение и git status, но всегда спрашивать перед git push и rm.
  • env — переменные окружения для сессий.
  • hooks — автоматические действия на события (см. модуль 9).

Пример фрагмента settings.json, который разрешает безопасные read-only команды без лишних подтверждений:

{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(ls:*)",
      "Read(*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push:*)"
    ]
  }
}
СоветМеньше лишних вопросов

Если Claude Code слишком часто спрашивает разрешение на однотипные безопасные команды, добавьте их в allow. Это и удобнее, и не притупляет вашу бдительность к действительно важным подтверждениям. Существует даже специальная команда, которая анализирует ваши прошлые сессии и предлагает безопасный allowlist.

Slash-команды

Внутри Claude Code много встроенных команд с префиксом /: /help, /login, /model, /clear (очистить контекст), /config и другие. Скоро вы научитесь добавлять свои slash-команды.

Модуль 8. Скиллы (Skills): самый недооценённый рычаг

Чему вы научитесь: понимать, что такое скиллы, как они устроены, как писать свои и какие скиллы максимально полезны.

Зачем это нужно

Скилл — это упакованная экспертиза, которую агент подгружает, только когда она нужна. Вместо того чтобы каждый раз объяснять «как мы делаем код-ревью» или «как собирать docx», вы один раз оформляете это как скилл — и агент применяет его автоматически в подходящий момент. Это и масштабирование знаний, и экономия контекста (полная инструкция грузится по требованию, а не висит всегда).

Как устроен скилл

Скилл — это папка с файлом SKILL.md. Минимальная структура:

my-skill/
├── SKILL.md          # описание + инструкции
├── reference.md      # (опц.) доп. материалы, грузятся по необходимости
└── scripts/          # (опц.) вспомогательные скрипты

SKILL.md начинается с frontmatter — метаданных в начале файла:

---
name: code-review
description: Проводит ревью изменений по пяти осям — корректность, читаемость, архитектура, безопасность, производительность. Использовать перед merge.
---

# Как проводить ревью

1. Получи diff текущих изменений.
2. Пройди по каждой оси...
(далее подробные инструкции)

Ключевой механизм — прогрессивное раскрытие (progressive disclosure). В контексте агента всегда висит только короткое description. Когда задача совпадает с описанием, агент читает полный SKILL.md, а при необходимости — и дополнительные файлы. Так вы можете иметь сотни скиллов, не раздувая контекст.

Заметкаdescription решает всё

Агент выбирает скилл по полю description. Пишите его как «когда применять» (триггеры), а не только «что делает». Хорошо: «Использовать перед коммитом для проверки на уязвимости при работе с пользовательским вводом». Плохо: «Скилл безопасности».

Где живут скиллы

  • Личные — ~/.claude/skills/<имя>/SKILL.md.
  • Проектные — .claude/skills/<имя>/SKILL.md (коммитятся, общие для команды).
  • Из плагинов — скиллы могут приходить пакетами-плагинами.

Откуда брать готовые скиллы и как их ставить

Скиллы бывают трёх источников:

  1. Свои — вы пишете их сами (см. ниже), кладёте в ~/.claude/skills/ (личные) или .claude/skills/ (проектные).
  2. Из плагинов/маркетплейса — целые наборы скиллов приходят пакетом-плагином. Подключение плагинов делается через настройки Claude Code (команда управления плагинами/маркетплейсом внутри интерфейса — см. /help, раздел про plugins). После подключения набора его скиллы становятся доступны автоматически.
  3. Из репозиториев — многие наборы лежат на GitHub: клонируете и кладёте папки скиллов в каталог скиллов.

Проверить, какие скиллы сейчас видит агент, можно через /help и встроенные команды управления; в начале сессии полезен мета-скилл using-agent-skills — он помогает агенту выбрать подходящий скилл под задачу.

Must-have наборы под цикл разработки (категориями, как в таблице ниже): набор инженерных скиллов (spec, plan, build, test, review, debugging, security, context-engineering), наборы для дизайна и UX-исследований, академический набор для письма и ревью. Начните с инженерного набора — он закрывает 80% задач разработки.

СоветСначала научите агента находить скиллы

Прежде чем ставить десятки скиллов, подключите мета-скилл using-agent-skills. Иначе агент может «не догадаться» применить нужный скилл — а с ним он в начале сессии сам соотносит задачу с доступными скиллами. Это разница между «скиллы лежат» и «скиллы работают».

В экосистеме Claude Code есть готовые наборы скиллов под весь цикл разработки. Вот категории, которые стоит освоить в первую очередь:

СоветМета-навык

Есть скилл «using-agent-skills» — он учит сам агент находить и применять нужные скиллы. Это «оркестратор скиллов»: с ним агент в начале сессии понимает, какой скилл подходит под текущую задачу.

Пишем свой скилл: пошагово

Допустим, вы хотите скилл «оформление PR по нашему шаблону».

  1. Создайте папку: mkdir -p .claude/skills/our-pr.
  2. Создайте SKILL.md:
---
name: our-pr
description: Оформляет описание pull request по нашему корпоративному шаблону — заголовок, контекст, изменения, тестирование, риски. Использовать при подготовке PR.
---

# Шаблон описания PR

Заполни разделы:
## Что изменилось
## Зачем
## Как тестировалось
## Риски и откат

Правила: заголовок в повелительном наклонении, до 70 символов; ссылайся на тикет.
  1. Перезапустите Claude Code. Теперь при подготовке PR агент сам применит шаблон.
ЗаметкаСкиллы работают и в API

Скиллы — это не только про CLI. Через Anthropic API можно подключать как готовые скиллы Anthropic (создание .pptx, .xlsx, .docx, .pdf), так и собственные. Это понадобится нам в части про агентов вне CLI.

Модуль 9. Субагенты, хуки, slash-команды и MCP

Чему вы научитесь: настраивать субагентов внутри Claude Code, автоматизировать действия хуками, создавать свои slash-команды и подключать внешние инструменты через MCP.

Зачем это нужно

До сих пор у нас был один агент. Реальная сила — в разделении труда: главный агент-оркестратор раздаёт подзадачи специализированным субагентам, каждый со своим контекстом и (что важно для экономии) своей моделью. Хуки автоматизируют рутину, slash-команды упаковывают частые запросы, а MCP даёт агенту доступ к внешнему миру (БД, API, файлы).

Субагенты внутри Claude Code

Субагент — это отдельный агент с собственной ролью, системным промптом, набором инструментов и моделью. Главный агент вызывает его как инструмент, передаёт задачу, получает результат — и собственный контекст субагента не засоряет контекст оркестратора.

Субагенты описываются файлами в .claude/agents/<имя>.md с frontmatter:

---
name: explorer
description: Read-only поиск по кодовой базе. Использовать, когда нужно найти, где что лежит, без правок.
tools: Read, Grep, Glob
model: haiku
---

Ты — исследователь кода. Твоя задача — быстро находить нужные файлы и функции
и возвращать краткий вывод: пути и строки, без длинных дампов. Ничего не меняй.

Разберём поля:

  • name / description — имя и «когда применять». Оркестратор выбирает субагента по описанию.
  • tools — какие инструменты разрешены. Принцип минимума прав: исследователю — только чтение/поиск.
  • model — какая модель. Дешёвый haiku для массового поиска, opus для сложного рассуждения. Это и есть рычаг экономии: 90% объёма делает дешёвая модель.
СоветЗачем изолировать субагентов

У субагента отдельное «окно контекста». Когда вы поручаете ему «прочитать 30 файлов и вернуть вывод», все эти 30 файлов попадают в его контекст, а к оркестратору возвращается только короткий итог. Так главный агент остаётся «с ясной головой» и не упирается в лимит контекста.

Хуки (hooks)

Хук — это команда, которую Claude Code выполняет автоматически на определённое событие. Хуки настраиваются в settings.json. Типичные применения:

  • запускать линтер/форматтер после каждой правки файла;
  • запускать тесты перед коммитом;
  • показывать уведомление, когда агент закончил длинную задачу;
  • блокировать опасные команды.
ВажноХуки исполняет харнесс, не модель

Важная мысль: автоматизм «каждый раз, когда X — делай Y» реализуется именно хуком, а не «памятью» или вежливой просьбой к агенту. Память может быть проигнорирована; хук исполняется детерминированно харнессом (программной обвязкой вокруг модели — самим Claude Code; подробнее в части III). Если вам нужно гарантированное автоматическое поведение — это хук.

Свои slash-команды

Свою команду можно создать файлом в .claude/commands/<имя>.md. Содержимое — это промпт-шаблон, который подставится при вызове /<имя>. Пример /<deploy-check>:

Проверь готовность к деплою: прогоны линта и тестов, незакоммиченные изменения,
обновлён ли CHANGELOG. Дай вердикт go/no-go списком.

Теперь команда /deploy-check будет запускать этот сценарий. Так вы превращаете повторяющиеся ритуалы в одну команду.

MCP — Model Context Protocol

MCP — это открытый стандарт, по которому агент подключается к внешним инструментам и данным: базам, API, файловым хранилищам, поисковикам, трекерам задач. MCP-сервер «публикует» набор инструментов, а Claude Code их использует.

Зачем это нужно: без MCP агент ограничен файлами и shell. С MCP он может, например, читать ваш Postgres, создавать задачи в трекере, искать в интернете, работать с GitHub. Подключение MCP-сервера настраивается в конфигурации Claude Code (через claude mcp add или в settings), указывается команда/URL сервера и, при необходимости, токены.

ЗаметкаMCP в этой установке

В реальной рабочей установке Claude Code, для которой готовился курс, подключены целые наборы скиллов и плагинов (например, академические и дизайн-наборы), а также инструменты исследования (веб-поиск, веб-загрузка). Это показывает масштаб: один агент в терминале может иметь доступ к сотням скиллов и десяткам внешних инструментов одновременно.

Плагины

Плагин — это пакет, который приносит сразу группу скиллов, субагентов, команд и MCP-серверов. Вместо ручной настройки каждого элемента вы подключаете плагин и получаете готовый набор под задачу (например, набор для академического письма или для дизайн-исследований).

Как это складывается вместе

Представьте рабочий процесс: вы даёте задачу оркестратору (Opus). Он включает plan mode, изучает проект (через субагента-исследователя на Haiku), пишет спецификацию (скилл spec), реализует инкрементально (скилл build), на каждом сохранении файла хук гоняет линтер, перед коммитом ваша slash-команда /deploy-check проверяет готовность, а данные для проверки агент берёт из БД через MCP. Это уже не «чат» — это конвейер.
